# 已知问题

## 当前限制

- 已支持真实视频元数据读取和关键帧抽取，但尚未接入独立 ASR 引擎。
- 如果视频没有字幕画面，也没有同名 `.txt`、`.srt` 或 `.vtt` sidecar，吐槽语句主要依赖 LLM/VLM 对关键帧和文件名的推断。
- LLM/VLM 调用默认走 Right Codes Codex Pro，模型 `gpt-5.5`；Gemini 仅作为 `compare-llm` 对比路径保留。
- 九帧图片生成默认走 Right Codes draw，模型已按要求切换为 `gpt-image-2`；如果本地设置了 `RIGHT_CODES_DRAW_MODEL`，会覆盖源码默认值。
- `generate:frames -- --no-draw` 会生成本地占位 sprite，只用于验证目录、manifest、绿幕去底、切帧和校验链路，不代表正式 AI 美术；正式视觉验收请使用 `_real` 包或 `--strict-draw`。
- 真实 draw 调用可能受模型耗时或 Cloudflare 超时影响；脚本默认优先走流式 `v1/chat/completions`，失败后会记录原因并回退占位，`--strict-draw` 会改为失败即中断。本轮 `anim_video_5f1f2057_real` 与 `anim_p1_ab3ea18b_real` 已通过真实 `gpt-image-2` 验收。
- 强 IP 视频默认走 `faithful` 忠实模式：第一轮保留精确角色名和原 IP 特征；如果图像模型因相似性安全规则拒绝生成，会第二轮去掉精确角色名但保留光剑、长耳、长袍、兜帽等视觉要素重试一次；仍失败才中断并记录原因。只有显式使用 `--safe-remix` 时才做更大幅度原创化改编。
- 未设置 `CODEX_PRO_API_KEY` 或 `RIGHT_CODES_API_KEY` 时，`understand:video` 会使用启发式 fallback；这只能验证结构和文件链路，不能替代真实 `gpt-5.5` 视频理解。
- 自动找主体已经作为生成阶段的离线后处理接入，可以过滤常见邻格碎片并统一帧锚点；但如果图像模型输出的九宫格严重跨格、角色本体被切断、多个主体粘连，后处理不能凭空补回缺失内容，需要通过 prompt 安全边距、降低画面复杂度或重新生成解决。

## 对游戏加载的影响

- 不影响第一版加载。`manifest.json`、`analysis_report.json`、`package_status.json`、`assets/` 和 `frames/` 均已生成。
- `package_status.json.validation.can_load_in_game` 为 `true` 时，游戏窗口可以直接加载。
- 素材缺失或模型调用失败时，包内仍保留 fallback，游戏可按白模继续运行。
- 游戏运行时不需要自动找主体。应直接读取 `frames/` 内固定尺寸 PNG，并按动画 manifest 的 `render_origin` 渲染角色或投掷物，避免每帧按左上角对齐造成偏移。

## 下一步建议

- 接入真实 ASR，把音频台词加入 `transcript_candidates`。
- 接入 OCR 或字幕提取，增强吐槽语句稳定性。
- 将游戏窗口默认加载目标切到 `_real` 包进行联调，确认 3x3 角色动作表、1x4 投掷物表和各帧 PNG 的加载方式。
