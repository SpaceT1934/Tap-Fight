# 场景资产生成扩展方案

更新时间：2026-06-09

## 2026-06-09 实现状态

第一阶段已经接入为可选开关，默认不启用，旧流程保持不变。

新命令开关：

```powershell
.\package_video.ps1 `
  -Video "<视频路径>" `
  -ThemeId "<theme_id>" `
  -GenerateStageAssets `
  -DrawConcurrency 2 `
  -DrawRetries 2 `
  -DrawTimeoutMs 300000
```

直接导入 Tap-Fight：

```powershell
.\package_video.ps1 `
  -Video "<视频路径>" `
  -ThemeId "<theme_id>" `
  -GenerateStageAssets `
  -ExportTapFight `
  -TapFightDir "<Tap-Fight目录>" `
  -StageTemplate "office_battle_001"
```

开关边界：

- 不传 `-GenerateStageAssets`：仍然只生成角色、投掷物、技能语句；Tap-Fight 场景继续从 `-StageTemplate` 复制。
- 传 `-GenerateStageAssets`：理解结果会记录 `scene_assets`，图片生成阶段额外生成背景、平台、危险物。
- `-StructureOnly -GenerateStageAssets` 只用于结构测试，会输出本地 fallback 图，不代表真实视觉质量。
- 真跑素材时默认严格 draw；没有 key 或 draw 失败会中断，不会静默产出假资产。

尺寸硬约束：

- `background.png`：固定 `1080x1920`，竖版 9:16。
- `platform_left.png`：固定透明底 `58x54`。
- `platform_mid.png`：固定透明底 `128x54`。
- `platform_right.png`：固定透明底 `58x54`。
- `hazard_debris.png`：固定透明底 `48x48`。
- 现有部分旧模板背景是 `720x1280`，比例同样是 9:16，可以被游戏加载；新生成资产统一使用 `1080x1920` 作为更清晰的固定输出规格。

输出位置：

```text
theme_packs/<themeId>/assets/stage/background.png
theme_packs/<themeId>/assets/stage/platform_left.png
theme_packs/<themeId>/assets/stage/platform_mid.png
theme_packs/<themeId>/assets/stage/platform_right.png
theme_packs/<themeId>/assets/stage/hazard_debris.png
```

导入 Tap-Fight 时会复制到游戏当前已经读取的根目录文件名：

```text
client/public/theme_packs/<themeId>/background.png
client/public/theme_packs/<themeId>/platform_left.png
client/public/theme_packs/<themeId>/platform_mid.png
client/public/theme_packs/<themeId>/platform_right.png
client/public/theme_packs/<themeId>/hazard_debris.png
```

导出优先级：

1. 如果主题包里存在 `assets/stage/<asset>`，优先使用生成版。
2. 如果缺失，继续从 `-StageTemplate` 复制模板版。
3. 技能特效图片接口尚未接入游戏端，仍保持 Canvas 动态效果。

## 目的

当前视频管线已经能从输入视频生成两个对抗角色、角色动作帧、投掷物、技能吐槽语句和结构化分析数据。游戏运行时的主题资产实际上分成两类：

1. 视频生成资产：角色动画、投掷物、技能语句、结构化报告。
2. 舞台模板资产：背景、平台、危险物等固定场景图。

下一步扩展目标是把第 2 类也逐步纳入生成流程，让一个视频主题包不仅换角色，也能生成同一风格的背景、平台、场景危险物和技能视觉特效。

## 当前运行时读取的资产

Tap-Fight 当前从 `client/public/theme_registry.json` 找到 `themeId` 和视频路径，再读取：

- `client/public/theme_packs/<themeId>/manifest.json`
- `client/public/theme_packs/<themeId>/animation_preview/p1_animation_manifest.json`
- `client/public/theme_packs/<themeId>/animation_preview/p2_animation_manifest.json`
- `manifest.projectile` 指向的投掷物帧，旧包回退到 `animation_preview/frames/p2_projectile/projectile_0.png`

游戏已经会读取这些场景文件，文件名目前是固定约定：

- `background.png`
- `platform_left.png`
- `platform_mid.png`
- `platform_right.png`
- `hazard_debris.png`

相关代码位置：

- `Tap-Fight/client/src/main.ts` 的 `loadThemePackage(...)`：加载 `manifest.environment.background`、三段平台图、危险物图、投掷物、角色动画。
- `drawBackground()`：绘制 `manifest.environment.background`。
- `drawPlatforms()`：绘制 `platform_left.png`、`platform_mid.png`、`platform_right.png`。
- `drawHazards()`：绘制 `hazard_debris.png`。
- `spawnVoiceSkillEffect()` / `drawVoiceSkillEffects()`：语音技能释放特效目前是 Canvas 动态圆环和粒子，不是图片资产。

## 当前生成器产出的资产

`package_video.ps1` 调用 `video-pipeline/src/index.mjs`，当前真实生成重点是：

- `assets/p1_sprite_sheet.png`
- `assets/p2_sprite_sheet.png`
- `frames/p1/*`
- `frames/p2/*`
- `assets/<player>_projectile_sheet.png`
- `assets/<player>_projectile.png`
- `frames/<player>_projectile/*`
- `p1_animation_manifest.json`
- `p2_animation_manifest.json`
- `p1_projectile_manifest.json` / `p2_projectile_manifest.json`
- `manifest.json`
- `analysis_report.json`
- `pipeline_result.json`
- `package_status.json`

`manifest.environment.background` 当前默认指向 `assets/background.png`，但在导出 Tap-Fight 时，`copyTapFightStageAssets(...)` 会从 `-StageTemplate` 对应的预制主题里复制 `background.png`、平台和危险物到最终游戏目录。

这说明：当前主题包中虽然可能有 `assets/background.png`，但进入游戏时主要还是预制舞台模板在生效。

## 当前边界

已经生成：

- 两个角色的动作九帧。
- 远程/投掷角色的投掷物四帧。
- 技能吐槽语句。
- 风格证据、角色/武器/投掷物描述、结构化报告。

已经被游戏支持，但主要靠模板复制：

- 背景图。
- 平台左中右三段。
- 危险物碎片。

游戏已有表现，但还没有图片资产接口：

- 语音技能眩晕特效。
- 技能释放粒子。
- 眩晕提示环。

暂不建议扩展到玩法层：

- 平台碰撞布局。
- 伤害判定。
- 联机规则。
- 技能数值。

这些属于游戏机制，不属于“素材生成与导入”边界。

## 可扩展性判断

可以扩展，而且第一阶段不需要改游戏玩法。

最稳的方式是先让视频管线生成与当前游戏约定同名的场景图片：

- `background.png`
- `platform_left.png`
- `platform_mid.png`
- `platform_right.png`
- `hazard_debris.png`

导出到 Tap-Fight 时，如果主题包里有生成版场景资产，就优先复制生成版；没有时继续回退到 `-StageTemplate`。这样旧主题、模板主题、生成主题都能共存。

技能特效建议作为第二阶段，因为当前游戏没有从 manifest 加载技能特效贴图。需要加一个可选字段，例如：

```json
{
  "effects": {
    "voice_skill": {
      "type": "radial_sheet",
      "image": "effects/voice_skill_ring_sheet.png",
      "frames": 4,
      "fallback": "canvas_ring"
    }
  }
}
```

游戏端加载不到这个字段时，继续使用现在的 Canvas 圆环和粒子。

## 推荐分阶段方案

### 第一阶段：生成场景图，不改游戏代码

新增生成器能力：

- 从视频理解结果里提取 `scene_brief`：世界观、材质、色彩、光照、空间关键词、禁用元素。
- 调用图片生成模型生成：
  - 竖版 9:16 背景图，匹配 Tap-Fight `540x960` 竖屏 canvas。
  - 透明底平台左端、中段、右端。
  - 透明底危险物碎片。
- 做基础质量检查：
  - 背景必须是非透明、横版、无大字、无角色占画面中央。
  - 平台和危险物必须有透明通道。
  - 平台中段左右可平铺，不能有明显主体人物。
- 只有质量不合格时才重画。

导出逻辑修改：

- `copyTapFightStageAssets(...)` 改成优先从主题包复制同名资产。
- 如果主题包缺某个资产，再从 `-StageTemplate` 复制。
- `background.png` 不再被模板无条件覆盖。

建议新增命令参数：

- PowerShell：`-GenerateStageAssets`
- Node：`--generate-stage-assets`
- 可选：`--stage-generation-mode template|generated|hybrid`

默认建议先用 `hybrid`：生成成功就用生成资产，失败则回退模板。

### 第二阶段：生成技能特效贴图，小幅改游戏加载

新增生成资产：

- `effects/voice_skill_ring_sheet.png`
- 可选 `effects/stun_burst.png`
- 可选 `effects/cast_spark.png`

新增 manifest 字段：

- `effects.voice_skill`
- `effects.stun`
- `effects.cast`

Tap-Fight 改动：

- `loadThemePackage(...)` 根据 manifest 加载 effects 图片。
- `drawVoiceSkillEffects(...)` 优先绘制生成贴图。
- 加载失败时仍使用当前 Canvas 圆环。

这一步不改变技能触发逻辑，也不改变眩晕时长。

### 第三阶段：可选生成舞台布局数据

如果后续确实需要“每个视频生成不同平台布局”，可以增加：

```json
{
  "stage_layout": {
    "platforms": [],
    "hazards": []
  }
}
```

但这会影响碰撞、平衡和联机同步，建议最后做，并且需要游戏同学一起验收。

## 推荐实现落点

生成器：

- `video-pipeline/src/index.mjs`
  - 增加 CLI 参数解析：`--generate-stage-assets`。
  - 增加 `buildSceneAssetPrompts(...)`。
  - 增加 `generateStageAssets(...)`。
  - 增加 `postprocessStageAsset(...)` 或轻量检查函数。
  - 修改 `exportTapFightTheme(...)` / `copyTapFightStageAssets(...)`，让生成资产优先于模板。

入口脚本：

- `package_video.ps1`
  - 增加 `[switch]$GenerateStageAssets`。
  - 传给 Node：`--generate-stage-assets`。

游戏端第二阶段：

- `Tap-Fight/client/src/main.ts`
  - `loadThemePackage(...)` 加载 `manifest.effects`。
  - `spawnVoiceSkillEffect(...)` 记录 effect image key。
  - `drawVoiceSkillEffects(...)` 优先绘制贴图，失败时走当前 Canvas。

## 风险和验收

主要风险：

- 背景生成容易把角色或文字画进去，需要 prompt 和检查约束。
- 平台图需要透明底和可平铺，否则游戏里会显脏或断裂。
- 技能特效如果太复杂，可能遮挡角色和血条。
- 生成完整场景会增加 draw 调用次数，需要默认可关闭。

验收标准：

- 不开 `-GenerateStageAssets` 时，旧流程完全不变。
- 开启后，Tap-Fight 目录里能看到生成版 `background.png` 和平台/危险物。
- 缺失或质量不合格时自动回退 `-StageTemplate`，不能产出假占位图冒充真实资产。
- `npm run validate` 通过。
- Tap-Fight `npm run build` 通过。

## 结论

这条扩展线是可行的。最合适的路线不是一次性改玩法，而是先把“场景视觉资产”纳入视频生成 ThemeAssetPackage：

1. 生成背景、平台、危险物。
2. 用同名文件兼容当前 Tap-Fight。
3. 导出时生成资产优先，模板兜底。
4. 再用可选 manifest 字段扩展技能特效贴图。

这样可以把“视频生成游戏资产”的展示从角色生成扩展到完整主题生成，同时保持现有游戏和旧主题包继续可用。

## 2026-06-09 平台生成优化

Tap-Fight 运行时仍然读取三个固定平台文件：

- `platform_left.png`：`58x54`
- `platform_mid.png`：`128x54`
- `platform_right.png`：`58x54`

这三个文件不是三种玩法平台，而是同一个矩形平台的视觉九宫格式切片：左端帽、中间拉伸段、右端帽。为了减少生成时间并保持材质连续，视频生成端不再分别调用图片模型生成三张平台图，而是：

1. 让图片模型生成一个统一平台条 `platform_strip`。
2. 后处理归一化为 `244x54`。
3. 按 `58 + 128 + 58` 固定切成 `platform_left.png`、`platform_mid.png`、`platform_right.png`。

最终 ThemeAssetPackage 和 Tap-Fight 导入结构不变，`manifest.stage_assets.files` 仍只暴露游戏实际读取的三个平台文件。
